# ProbeFlow Physics & Math Correctness Review
Date: 2026-05-27
Reviewer: physics-aware agent (parallel deep review pass)

## Summary
- 17 findings: S0=0, S1=4, S2=10, S3=3
- Top 3 concerns:
  1. `fft_magnitude` subtracts the wrong mean when an ROI mask is used: it averages over zero-padded outside-ROI pixels, leaving residual DC and creating a low-k ring artifact in the magnitude spectrum (S1).
  2. `rotate_arbitrary` uses `cval=NaN` with bilinear interpolation, so every interior pixel within one pixel of the boundary becomes NaN through interpolation contamination — the rotated image loses a fuzzy halo of valid data (S1).
  3. `quantize_bit_depth` (new 8/16-bit Image Type) derives `vmin`/`vmax` from raw data extrema, so a single bad/tip-crash pixel wastes most of the levels on noise outside the meaningful signal range; downstream metrology of small heights loses far more precision than the docstring "~3.9 pm per level for 1 nm Z-range" suggests (S1).

## Findings

### 1. [S1] `fft_magnitude` subtracts a biased mean when masking a non-rectangular ROI
**Location**: `probeflow/processing/filters.py:425-435`
**Problem**: The order of operations is (a) fill NaN with `nanmean(crop)`, (b) zero pixels outside the ROI mask, (c) subtract `crop.mean()`. After step (b) the array contains real values inside the ROI and exactly 0.0 outside. `crop.mean()` is therefore `(sum_inside_ROI) / total_pixels`, i.e. the inside-ROI mean attenuated by the ROI fill fraction. Step (c) removes only that fraction of the true DC component from inside-ROI pixels, and pushes the outside-ROI pixels to a negative bias. The boundary discontinuity that results adds a strong low-frequency cross / ring to the magnitude spectrum.
**Why it matters**: Users running an elliptical or polygonal ROI FFT (Bragg-peak analysis, periodicity hunting) see spurious low-k features and reduced peak contrast. For small ROIs (low fill fraction), the spurious DC dominates.
**Suggested fix**: Compute `inside_mean = crop[mask].mean()` *before* zeroing, subtract that from inside-ROI pixels only, and keep outside-ROI at zero. The subsequent window already tapers the boundary so a residual step at the boundary is partially suppressed, but the explicit DC removal must be the inside-ROI mean.

### 2. [S1] `rotate_arbitrary` contaminates valid pixels with NaN via interpolation
**Location**: `probeflow/processing/geometry.py:518-521`
**Problem**: The call is `_ndimage_rotate(a, angle, reshape=True, order=order, mode='constant', cval=np.nan)`. With `order=1` (bilinear), every output pixel within one pixel of the boundary samples one (or more) NaN neighbours, and a NaN contribution with any non-zero weight propagates as NaN. The effective valid region shrinks by ~1 pixel all around the rotated content, and at oblique angles the loss can stretch deeper along the new edges.
**Why it matters**: After a 30° rotation followed by analysis (lattice extraction, FFT, profile sampling), the boundary band is silently dropped or seeded into downstream measurements. Repeated rotations compound the loss.
**Suggested fix**: Use the same NaN-aware pattern as `affine_lattice_correction`: temporarily fill NaN with the finite mean, run the rotation with `cval=temp_fill` (or `mode='reflect'`), then mask the output to NaN only where the inverse map falls outside the input bounds (use the input nan_mask transported through the same rotation with `order=0`).

### 3. [S1] `quantize_bit_depth` levels are spread over raw data extremes, not the meaningful display range
**Location**: `probeflow/processing/geometry.py:641-678` (new in commit cafb10d)
**Problem**: `vmin = arr[finite].min(); vmax = arr[finite].max()` then `(n_levels-1)/(vmax-vmin)` sets the level spacing. A single saturated row, tip-crash spike, or large step edge sets vmax/vmin to extreme values, so the 8-bit "256 levels" are stretched across, say, 50 nm even though the surface of interest spans only 0.2 nm. The effective precision on the real signal collapses by 1–2 orders of magnitude, with no warning to the user. The docstring's "~3.9 pm per level for 1 nm Z-range" example silently assumes the data range equals the feature range.
**Why it matters**: 8-bit quantization is a typical pre-export step before saving uint8 PNG/TIFF or feeding ImageJ-compatible workflows. If a user crashes the tip mid-scan, then runs Image > Type > 8-bit, terrace-step measurements after that step show only ~8 distinct height bands instead of ~256.
**Suggested fix**: Either (a) accept explicit `vmin/vmax` arguments and have the GUI pass the current display range (which already excludes percentile-tail outliers), or (b) default to percentile bounds (e.g. `np.percentile(finite, [0.1, 99.9])`) and document the clipping. Clip-out values to vmin/vmax (or NaN) rather than letting them define the scale.

### 4. [S1] Autocorrelation periodicity is biased toward shorter periods
**Location**: `probeflow/analysis/line_periodicity.py:194-201`, `_autocorrelation_method`
**Problem**: `np.correlate(z, z, mode="full")` gives the biased estimator: overlap shrinks linearly with lag, so the un-normalised ACF amplitude decays as `(n - k)/n` even for a perfectly periodic signal. Dividing by `zero_lag_val` (line 197) does not undo this — it only normalises the zero-lag value to 1. The peak detector then sees later-lag harmonics suppressed, biasing the chosen period toward the first (shortest) peak.
**Why it matters**: For a profile carrying ≥3 periods, the first ACF peak is correctly recovered, but the ambiguity check at lines 248-255 compares to higher-lag peaks whose heights have been artificially suppressed, so spurious shorter periods can pass the prominence threshold. Reported uncertainty (FWHM in lag) is also biased low.
**Suggested fix**: Divide `ac_pos` by `(n - np.arange(1, n))` to get the unbiased ACF (or use `scipy.signal.correlate` with the explicit lag-normalisation pattern). The denominator should not go below ~`n/8` to avoid amplifying noise at extreme lags — clip the search range accordingly.

### 5. [S2] `fft_magnitude` k-axes are not normalised for window power, but `find_bragg_peaks_in_q_annulus` consumes the same magnitude
**Location**: `probeflow/processing/filters.py:460-470` (the in-source NOTE acknowledges this), used by `find_bragg_peaks_in_q_annulus` `filters.py:750-834`
**Problem**: The magnitude returned by `fft_magnitude` is `|F|` with no compensation for the Hann/Tukey window's coherent gain (~0.5 for Hann in 1-D, ~0.25 in 2-D outer product). The peak detector and score thresholds in `find_bragg_peaks_in_q_annulus` (lines 781, 794, 811) operate on `log1p(|F|)`. Because the relative attenuation depends on window choice and image size (since `np.hanning(N)` includes the zero endpoint, the energy of `np.outer(hann, hann)` is ~0.213, not the textbook 0.25), the absolute thresholds (`prominence`, `min_separation_frac`, the MAD-based `_origin_streak_angles`) are window-sensitive in a way the user cannot see.
**Why it matters**: Switching the window from Hann to Tukey or "none" changes both `expected_count` and `_is_compact_peak` decisions for the same physical scan. This is a reproducibility hazard.
**Suggested fix**: Either normalise the magnitude by the window coherent gain (`win2d.sum() / (Nx*Ny)` for amplitude) and remove the in-source NOTE caveat, or have `find_bragg_peaks_in_q_annulus` work on a *score* (e.g. z-score within the annulus) rather than absolute log magnitude.

### 6. [S2] `np.hanning(N)` zero-endpoint window depresses the boundary further than expected
**Location**: `probeflow/processing/filters.py:73-74, 82, 443, 454`; `probeflow/analysis/line_periodicity.py:327` (`np.hanning(n)`)
**Problem**: `np.hanning(N)` is the symmetric Hann window with explicit zero endpoints (`hann[0] == hann[N-1] == 0`). For an N-point DFT, you want the *periodic* window (`np.hanning(N+1)[:-1]` or `signal.windows.hann(N, sym=False)`) so that the windowed segment is genuinely periodic. The current choice subtracts coherent gain by exactly suppressing two pixels (which doesn't matter at large N but is significant for small ROIs / cropped FFTs) and introduces a small bias in the bin-center frequency mapping.
**Why it matters**: For small ROIs (e.g. 32×32 Bragg crops) and short 1-D line profiles, peak position and amplitude in the spectrum shift by ~1/N. Cumulative effect on `predicted_bragg_radius` vs measured radius can mis-calibrate piezo correction by ~1%.
**Suggested fix**: Switch to `scipy.signal.windows.hann(N, sym=False)` (or equivalent) in FFT-magnitude, autocorrelation, periodicity, and notch-filter codepaths. Keep `np.hanning` only where a strictly symmetric window is required (rare in DFT contexts).

### 7. [S2] `fourier_filter` radial cutoff uses normalised-to-half-axis coordinates, not Nyquist-fraction in cycles/pixel
**Location**: `probeflow/processing/filters.py:88-97`
**Problem**: The radial coordinate is `R = sqrt(((x-cx)/cx)^2 + ((y-cy)/cy)^2)`, i.e. each axis is normalised to half-image. The cutoff is documented as "fraction of Nyquist [0, 1]". For square images this is approximately correct (the corner is `sqrt(2)` and the axis maximum is 1.0, which both correspond to Nyquist), but for rectangular images the normalisation distorts the radial mask into an ellipse in cycles/pixel space. A `cutoff=0.5` filter looks at radius 0.5 of half-axis along both directions, which is a different physical k along x vs y when Nx ≠ Ny.
**Why it matters**: On rectangular STM scans (common when only one fast-scan dim is reduced) the "isotropic" low-pass is actually anisotropic in q-space. Features along the long axis are smoothed differently than along the short axis.
**Suggested fix**: Build the radial mask in true frequency units: `qx = fftshift(fftfreq(Nx))`, `qy = fftshift(fftfreq(Ny))` (in cycles/pixel), then `R = sqrt(qx[None,:]**2 + qy[:,None]**2) / 0.5` so the unit "1.0" lines up with Nyquist.

### 8. [S2] `compute_pair_correlation` has no edge correction — large-r bins are biased low
**Location**: `probeflow/analysis/pair_correlation.py:104-109` (acknowledged in `message` at line 122)
**Problem**: Standard `g(r) → 1` at large r normalization assumes infinite domain. With a finite ROI, the number of available pair-distances at radius r decreases as r approaches the half-extent (since the shell area is partially outside the ROI). The current normalisation divides by `2π r dr`, which is the full annulus area. Result: the histogram bias bias drops below 1 systematically as r grows, even for a perfect Poisson process.
**Why it matters**: Users reading `g(r)` to identify ordering (where features peaks above 1) or anti-correlation (where features dip below 1) are misled at scales near `0.5 * sqrt(roi_area_m2)` (the default `r_max_m`). The "first_peak_m" detector still works, but interpretation of the asymptote is wrong.
**Suggested fix**: Implement Ripley's edge correction or the simpler "available area" correction: divide each bin by the fraction of the shell that lies inside the ROI. For rectangular ROI this can be done analytically; for arbitrary ROI use a Monte-Carlo estimate. Even a simple "translate-and-count" baseline (number of random-point shells that fit inside the ROI) would remove most of the bias.

### 9. [S2] `_subtract_scanline_background` `preserve_level` adds back a *grand mean* that doesn't preserve per-row median
**Location**: `probeflow/processing/background.py:285-313`
**Problem**: After subtracting `fitted_profile[:, None]` row-wise, the code adds back `_finite_median(background, default=0.0)` — i.e. the median (or mean) of the *entire 2-D fitted background image*. This shifts every pixel by a single scalar. The function name and docstring imply per-row level preservation, but it's a global shift. For nonlinear fits (e.g. piezo_creep, sqrt_creep) the per-row baseline varies; subtracting them and adding back a single median produces an image whose rows are not at the original median heights — they're at heights `original_median - row_fit_value + global_fit_median`, which still has the row-to-row trend of the fit if the fit isn't perfect.
**Why it matters**: For metrology where users want to compare absolute heights between an STM scan-line corrected image and its original (e.g. step heights, terrace altitudes), the absolute z reference drifts in a model-dependent way. Users assume "preserve_level='median'" means the median of every row stays put.
**Suggested fix**: If the intent is per-row level preservation, subtract `fitted_profile[:, None]` then add back `np.median(fitted_profile)`. If the intent is "median of the full image stays put", document that explicitly and rename to avoid confusion with row-wise leveling.

### 10. [S2] Periodicity FFT in `analysis.py:measure_periodicity` is not normalised, so "strength" is window- and size-dependent
**Location**: `probeflow/processing/analysis.py:201-265`
**Problem**: `power = |fft2(arr * hanning_window)|^2`. The returned `strength` field is the raw power in arbitrary units. There's no normalisation by image energy, window correction, or pixel count. Comparing strengths across scans (or across cropped ROI vs full image) is meaningless. Additionally, the "peak suppression" uses `Ny - py, Nx - px` for the conjugate (line 257), which is correct for *even* dimensions but off by one for odd dimensions (the FFT-shifted symmetry centre of an N-point transform is at `floor(N/2)`, not `N/2` for odd N).
**Why it matters**: (a) Users sorting periodicities by "strength" across panels see inconsistent rankings. (b) For odd-dimensioned scans (rare but possible after some crops), the mirror-suppression misses the symmetric partner by one pixel, so the same physical peak can be reported twice as two adjacent peaks with the same period but slightly different angles.
**Suggested fix**: (a) Normalise `power` by `(window2d.sum())**2 * arr.size` or report `power / DC_power` so the value is a relative spectral fraction. (b) Use `(Ny - py) % Ny` and `(Nx - px) % Nx` for conjugate indexing, or build a `mirror_map` once with `fftshift` symmetry semantics.

### 11. [S2] `line_periodicity._fft_method` "uncertainty" formula is a frequency-resolution lower bound, not a real σ
**Location**: `probeflow/analysis/line_periodicity.py:354-355`
**Problem**: `uncertainty_m = period_m**2 * freq_res` where `freq_res = 1.0 / line_length_m`. This is just the bin-width in period space at the dominant frequency — an absolute *resolution floor*, not a statistical uncertainty. The autocorrelation FWHM and the peak-spacing std (in the other two methods) at least scale with the noise. The FFT branch reports the same uncertainty for a perfect crystal and a noisy profile — both will only depend on the line length.
**Why it matters**: A user pasting "Period: 0.246 ± 0.012 nm" into a publication assumes ± is a measurement uncertainty.
**Suggested fix**: Either compute an SNR-weighted parabolic-interpolation uncertainty around the FFT peak (similar to the autocorrelation branch's quadratic refinement, then propagate), or relabel the field as `bin_width_m` and surface SNR separately so the user can decide.

### 12. [S2] `_polar_decompose` sigma flip on reflection produces a non-positive-semidefinite stretch
**Location**: `probeflow/analysis/lattice_distortion.py:194-200`
**Problem**: When `det(R) < 0`, the code flips one column of `U` *and* negates the last singular value (`sigma[-1] *= -1`). After this, `S = Vt.T @ diag(sigma) @ Vt` has a negative eigenvalue and is no longer positive-semidefinite. The "preserve image orientation" path then applies this S, which includes an axis flip in addition to anisotropic stretch — exactly the reflection the polar decomposition was supposed to absorb.
**Why it matters**: When the measured-vs-ideal map T has a reflection component (e.g. due to image orientation flip during measurement), applying the "stretch-only" S also flips one axis of the image. The user will see a mirrored output without warning. For well-behaved inputs this code path doesn't fire, but it's a latent trap.
**Suggested fix**: Use a standard polar decomposition that handles reflections by absorbing them into R (so R is improper-orthogonal with det=-1 if the input has a reflection, and S stays positive-semidefinite). Or: detect `det(T) < 0` early and raise (or warn) — measured lattices shouldn't be reflected relative to ideal in normal workflows.

### 13. [S2] `extract_lattice` chooses lattice basis from displacement clusters with `cluster_choice=1` arbitrarily
**Location**: `probeflow/analysis/lattice.py:253-301`
**Problem**: After clustering kNN displacements, candidates are sorted by `|v|` and the two smallest non-colinear vectors are picked as the primitive basis. For a hex lattice this naturally returns two basis vectors at 60° (or 120°). But there is no enforcement that the chosen pair forms a *primitive* cell rather than a centered/conventional cell, and no preference for the canonical reduced basis (shortest pair with |γ| as close to 60° or 90° as possible). For images with thermal/scan distortion, the two smallest vectors may not be primitive — a centred-rectangular cell can give two vectors at 90° that are √2 longer than the true primitive basis.
**Why it matters**: Downstream feature-lattice comparison and lattice correction use these vectors as ground truth. A non-primitive choice scales the (i, j) site grid by 2× along one axis, doubling apparent off-lattice points and halving occupancy.
**Suggested fix**: After choosing the two shortest non-colinear vectors, run a Lagrange-Gauss lattice basis reduction (or LLL in 2-D) to ensure the returned (a, b) is the shortest-vector basis. This is a small post-processing step that ensures canonicity.

### 14. [S2] `compute_correction` ill-conditioned tolerance scales with `|a||b|` not with the sin of the angle
**Location**: `probeflow/analysis/lattice_distortion.py:119-126`
**Problem**: The collinearity tolerance is `tol = 1e-6 * la * lb`. The determinant `det(M) = |a||b| sin(γ)`. So `det < tol` corresponds to `sin(γ) < 1e-6`, i.e. γ within ~6e-5 deg of 0 or 180. This is *extremely* tight. Real measurements with nearly-collinear (e.g. γ = 5°) measured vectors will pass the check but produce a numerically unstable correction matrix.
**Why it matters**: A user picks two near-collinear lattice points by mistake and gets a correction that maps the image into an extreme aspect-ratio strip without warning, instead of a clear error.
**Suggested fix**: Use `sin(angle_between(a, b)) >= 0.05` (≈3°) as the lower bound. Or equivalently `|det(M)| >= 0.05 * |a| * |b|`.

### 15. [S3] `apply_outlier_mask` "threshold" is in raw-MAD units, not σ-equivalent units
**Location**: `probeflow/spectroscopy/outliers.py:62-81`
**Problem**: `robust_scores` returns `dev / mad`, with no `1.4826` scaling (the standard MAD-to-σ multiplier for normally distributed data). A documented threshold of `6.0` (the default) therefore corresponds to ~9.0 σ, not 6.0 σ. This makes the outlier filter much less aggressive than a user familiar with z-score conventions would expect.
**Why it matters**: Defaults are deceptive; reproducibility across packages is poor. Note that `bad_lines._robust_scale` (line 72) *does* apply the 1.4826 factor, so the two MAD code paths in the same codebase disagree.
**Suggested fix**: Either apply the 1.4826 factor in `robust_scores` (and update the threshold default accordingly), or relabel the threshold as "MAD multiples" in the GUI and docstring to match the actual behaviour.

### 16. [S3] `quantize_bit_depth` rounds away from the original quantisation
**Location**: `probeflow/processing/geometry.py:672-677`
**Problem**: The round-trip `np.round((arr - vmin) * scale) / scale + vmin` introduces a fixed bias of ½ level per pixel for the upper half of the range (because `round` rounds halves to nearest-even). For an array of integers that already sits at level boundaries this is fine, but for floats the cumulative DC level is shifted by ~½ level relative to the original. Not a real-world issue (precision is ½ level by definition after quantisation), but the operation is not idempotent: `quantize(quantize(arr, 8), 8)` can produce slightly different values from `quantize(arr, 8)` if `vmin`/`vmax` change between calls (which they may, e.g. if NaN handling alters min/max).
**Why it matters**: Re-applying an 8-bit conversion to an already-8-bit image shouldn't move pixel values. Currently it can.
**Suggested fix**: Snap `vmin = math.floor(vmin / quantum)*quantum` and `vmax = math.ceil(vmax / quantum)*quantum` (where `quantum` is the natural unit), or short-circuit when the input is already at integer multiples of the level spacing. Lower-priority because the magnitude of the bias is ½ LSB.

### 17. [S3] `feature_points_to_csv` does not enforce that `pixel_size_x_nm` and `pixel_size_y_nm` are both supplied
**Location**: `probeflow/analysis/feature_finder.py:217-242`
**Problem**: The signature accepts independent x/y pixel sizes (good), but no consistency check, and the column header says `x_nm`/`y_nm` without per-row tagging of the conversion. If a downstream tool reads these columns and computes distance with `sqrt((Δx_nm)² + (Δy_nm)²)`, it gets the correct physical distance — but only because both columns have already been multiplied by their own px size. The CSV format itself is fine; the issue is that the default `pixel_size_x_nm=1.0, pixel_size_y_nm=1.0` silently treats output as nm even when the caller hasn't supplied calibration, producing numerically valid but physically meaningless "x_nm" values equal to the pixel index. No warning is emitted.
**Why it matters**: A user exporting CSV without thinking about pixel size sees plausible nm values that are actually pixel indices.
**Suggested fix**: Make the pixel sizes positional (no default) or have them default to `None` and skip the nm columns when both are None.
