# ProbeFlow IO & Instrument Format Robustness Review
Date: 2026-05-27 (written 2026-05-28 retry, sequential mode)
Reviewer: IO robustness agent (deep review pass)

## Summary
- 22 findings: S0=0, S1=5, S2=12, S3=5
- Top 3 concerns:
  1. `write_sxm_with_planes` (the low-level SXM writer that every higher-level path delegates to) has zero overwrite/source-collision protection — a caller that passes `out_sxm == src_sxm` corrupts the original instrument file silently (S1).
  2. Default ROI sidecar path collapses dotted Createc filenames (`A250320.191933.dat → A250320.rois.json`), so saving ROIs *silently* writes them to the wrong sidecar name (and collisions across files that share a date prefix) (S1).
  3. Createc `_detect_channel_count` overrides the header's declared `Channels=2` whenever payload trailing diagnostics push the float count past `4*Nx*Ny`, producing two synthetic backward planes that contain instrument telemetry, not real scan data (S1).

## Findings

### 1. [S1] SXM writer fast path bypasses overwrite/source checks via `write_sxm_with_planes`
**Location**: `probeflow/io/sxm_io.py:353-415` (`write_sxm_with_planes`); call site `probeflow/io/writers/sxm.py:117-121`; test usage `tests/test_sxm_io.py:165`
**Problem**: `write_sxm_with_planes` writes `out_sxm` unconditionally via `out_sxm.write_bytes(...)`. There is no existence check on the destination and no comparison against `src_sxm`. The higher-level `write_sxm` calls `check_overwrite` (against `scan.source_path`) and `check_output_available`, but `write_sxm_with_planes` is itself a documented public function (per the module docstring at line 20) and tests already call it directly. The function will happily overwrite the source file if `src_sxm == out_sxm`. The conservative-overwrite contract advertised in README is broken at the underlying writer level.
**Why it matters**: A simple bug in any caller that passes `out_sxm=src_sxm` corrupts the original instrument file. Even when called via `write_sxm`, the `check_overwrite(scan.source_path, out_path)` is skipped when `scan.source_path is None` (see writers/sxm.py:71-72), so an in-memory Scan (e.g. one constructed from CLI fixtures or tests) can land its output on top of any pre-existing file with no guard.
**Suggested fix**: Add at the top of `write_sxm_with_planes`: `if Path(src_sxm).resolve() == Path(out_sxm).resolve(): raise ValueError(...)` and `check_output_available(out_sxm, overwrite=overwrite)`. Thread the `overwrite` kwarg through. Make the function private or document that callers are responsible for collision checks.

### 2. [S1] Default ROI sidecar path drops the inner extension on dotted filenames
**Location**: `probeflow/io/roi_sidecar.py:16-19` (`default_roi_sidecar_path`)
**Problem**: `path.with_suffix("").with_suffix(".rois.json")` is the canonical save path. For typical filenames it works. For Createc files saved as `A250320.191933.dat`, `Path.with_suffix("")` removes only `.dat`, leaving `A250320.191933`. Then `Path("A250320.191933").with_suffix(".rois.json")` replaces the last extension-like component (`.191933`), producing `A250320.rois.json`. Verified: `python3 -c "from pathlib import Path; print(Path('A250320.191933.dat').with_suffix('').with_suffix('.rois.json'))"` outputs `A250320.rois.json`. The candidate-load list at lines 30-44 includes `path.parent / f"{path.stem}.rois.json"` so existing sidecars in the right place can be found, but new saves go to the wrong name.
**Why it matters**: Createc DAT files use the timestamp-style naming `Aymmdd.HHmmss.dat`. After editing ROIs and saving, the sidecar lands at `A250320.rois.json` — colliding across every scan that shares the same date prefix. The next file saved that day silently overwrites the prior file's ROIs.
**Suggested fix**: Use `path.parent / f"{path.stem}.rois.json"` directly. The `with_suffix` chaining is fundamentally wrong for filenames with dots.

### 3. [S1] Createc `_detect_channel_count` quietly disagrees with header `Channels`
**Location**: `probeflow/io/readers/createc_dat.py:325-358`
**Problem**: The function first tries `header_channels = Channels` and accepts it only when `payload_float_count >= needed`. But it never *constrains* the detected count to the header. So when the header says `Channels=2` and the payload contains `>= 4*Ny*Nx` floats (Createc routinely tail-pads files with telemetry — `ignored_tail_float_count` handling acknowledges this), the function falls through to `for n in (4, 2, 1)` and returns 4. Two synthetic "backward" planes are then created from whatever bytes follow the legitimate 2-plane payload.
**Why it matters**: Real 2-channel Createc STM scans with trailing diagnostics are misread as 4-channel scans. The "Z backward" and "Current backward" planes contain instrument metadata bits interpreted as float32. The downstream `Scan` reports four planes with the canonical layout; forward/backward blending produces visible noise; provenance records the file as 4-channel.
**Suggested fix**: When `header_channels > 0` and the payload size is consistent with that channel count (allowing some trailing tail), trust the header verbatim. Only fall through to the (4, 2, 1) probe heuristic when `Channels` is missing or zero.

### 4. [S1] Nanonis SPM reader silently coerces missing tip position to (0.0, 0.0)
**Location**: `probeflow/io/readers/nanonis_spec.py:202-203` and `_parse_header_float:434-441`
**Problem**: `_parse_header_float(hdr, "X (m)")` returns `0.0` on any failure (missing key, empty value, malformed float). The resulting `(0.0, 0.0)` position is indistinguishable from a real spectrum acquired at the scan origin (which is a common Nanonis convention for "centred"). The optional-variant `_parse_optional_header_float` (line 444) returns `None` on missing — and is used for `Bias>Bias (V)` — but not for the tip position. `SpecMetadata.position` is typed as `tuple[float, float]`, so downstream callers assume a numeric pair.
**Why it matters**: A user opening an old Nanonis spec file (or one written by third-party tooling) without `X (m)` / `Y (m)` keys sees their spectrum painted at the origin instead of being warned that the position is unknown. Multi-spectrum overlays show all such spectra stacked at one point.
**Suggested fix**: Use `_parse_optional_header_float` for X/Y, change `SpecMetadata.position` to `tuple[float | None, float | None]`, and update callers to handle the None case (e.g. by suppressing the overlay marker).

### 5. [S1] DAT-to-SXM writer ignores `SCAN_DIR='up'` on writeback
**Location**: `probeflow/io/writers/sxm.py:150-152` (`_undo_orient`) and `probeflow/io/converters/createc_dat_to_sxm.py:395-422` (`process_dat`)
**Problem**: When the source DAT has `ScanYDirec=0`, `construct_hdr` sets `SCAN_DIR='up'` in the SXM header (line 223). But neither `_undo_orient` (writers/sxm.py) nor `process_dat` apply a `np.flipud` before write. The reader's `orient_plane` (sxm_io.py:218-220) *will* flip vertically on the next read because the header says `SCAN_DIR='up'`. Net effect: the round-tripped image is flipped upside-down compared to the source DAT. The DAT-sourced reconstruction path is therefore wrong specifically for the less-common `ScanYDirec=0` files. The `write_sxm_with_planes` fast path at lines 402-409 of sxm_io.py *does* apply `np.flipud` for `SCAN_DIR='up'`, so SXM-sourced round trips are consistent — only the DAT-sourced path drops the flip.
**Why it matters**: A user converting a Createc DAT acquired with reverse Y scan direction to SXM gets a file that the SXM reader will display upside-down. Step heights are still correct, but overlays, lattice fits, and visual comparisons against the original DAT in the GUI are inverted.
**Suggested fix**: In `process_dat` (or in `_write_from_dat._undo_orient`), inspect the constructed SXM header's `SCAN_DIR` and apply `np.flipud` to every plane when it is `'up'`. Alternatively, force `construct_hdr` to always write `SCAN_DIR='down'` and pre-flip the planes when needed. Add a regression test fixture covering `ScanYDirec=0`.

### 6. [S2] `parse_header` in `probeflow/io/common.py` discards Latin-1 bytes silently
**Location**: `probeflow/io/common.py:22-31`
**Problem**: The generic Createc/Nanonis header parser decodes raw bytes with `decode("ascii", errors="ignore")`. Real Createc files contain Latin-1 `Å` (0xC5) in keys like `Dacto[Å]z` (per `docs/createc_dat_reader.md`) and may use it elsewhere in older vintages. With `errors="ignore"`, the `Å` byte is silently dropped, so a key parses as `Dacto[]z` instead of `Dacto[Å]z`. `_canonical_createc_header_key` normalisation in `createc_dat.py` never sees `[Å]`. The value cannot be located by either spelling.
**Why it matters**: `parse_header` is currently only used by tests and is unreferenced by production reader code (the production Createc reader uses its own latin-1-aware `_parse_createc_dat_header`). The risk is therefore a *latent inconsistency*: any new caller wired to `parse_header` would silently get worse parsing than the production reader. Severity is S2 because no production caller is affected today.
**Suggested fix**: Replace `decode("ascii", errors="ignore")` with the same `try ascii / except UnicodeDecodeError: latin-1` fallback used in `_parse_createc_dat_header`. Or delete the dead export.

### 7. [S2] RHK SM4 image payload dtype disambiguation ignores `data_size` header field
**Location**: `probeflow/io/readers/rhk_sm4.py:484-519` (`_decode_image_payload`); `data_size` parsed at line 290 but never consulted
**Problem**: The decoder picks dtype based on payload byte length: `n*2 → int16`, `n*4 → int32`, `n*8 → float64`. The page header already carries an explicit `data_size` field, but it is not consulted. For images where `len(payload)` ambiguously matches more than one branch (degenerate case: `x_size=0` or partially-corrupt headers), the first match wins. For SM4 files with non-standard payload sizes, the decoder either picks the wrong dtype or falls into the int16 fallback when int32 was intended. No warning is emitted distinguishing "raw int32 counts" from "compressed via zlib then int32" — both go through silently.
**Why it matters**: New RHK firmware variants or hand-edited fixtures can produce mis-typed arrays that pass shape checks but are off by `2^16` in scale. Combined with finding 22's unit-normalisation gap, the failure is silent.
**Suggested fix**: Use `header["data_size"]` as the authoritative bytes-per-pixel indicator; warn when length is consistent with multiple dtype choices.

### 8. [S2] `parse_sxm_header` collapses multi-paragraph values and silently drops blank lines
**Location**: `probeflow/io/sxm_io.py:78-113`
**Problem**: The parser does `" ".join(buf).strip()` for each `:KEY:` block, so multi-line values (e.g. `COMMENT` with paragraph breaks, `Z-CONTROLLER` with tab-separated rows) lose their line structure entirely. Blank lines inside a value are skipped silently (`if s: buf.append(s)`). Round-trip through `write_sxm_with_planes` (which preserves header *bytes* verbatim — good) hides this, but any downstream consumer using `parse_sxm_header` to read the comment or tabular Z-CONTROLLER block sees a single space-collapsed line.
**Why it matters**: Multi-paragraph user comments are flattened; tabular Z-CONTROLLER metadata becomes a single line that downstream parsers can't split by row.
**Suggested fix**: Preserve continuation lines verbatim with `"\n".join(buf)` and strip only trailing newlines. For round-trip safety, document the byte-copy guarantee of `write_sxm_with_planes` so downstream code does not re-emit from the parsed dict.

### 9. [S2] `sxm_data_info` parses by raw whitespace; channel-name spaces and partial rows silently drop
**Location**: `probeflow/io/sxm_io.py:139-170`
**Problem**: After `parse_sxm_header` collapses continuation whitespace, `sxm_data_info` does `raw.split()` and consumes 6-token chunks. Nanonis channel names with spaces are normally stored with underscores (and `_display_channel_name` un-translates them back), but the loop `for i in range(start, len(toks), 6)` silently drops a partial trailing row (line 158-159). A `DATA_INFO` table with one truncated final row loses that channel without raising. Combined with finding 8 (whitespace collapse), the entire schema is fragile to any variant header.
**Why it matters**: A hand-edited or vendor-variant header with a malformed final row silently drops a channel; the `Scan` then has fewer planes than the file actually contains, and the warning at sxm_io.py:274 ("payload contains N planes but DATA_INFO indicates M") fires backwards or not at all.
**Suggested fix**: Preserve the original line structure for `DATA_INFO` in `parse_sxm_header` (it's a tabular section, not a single value). Parse line-by-line. Emit a warning if a partial row is encountered.

### 10. [S2] `read_all_sxm_planes` silently truncates plane list and discards the truncation flag downstream
**Location**: `probeflow/io/sxm_io.py:256-293`; consumer `probeflow/io/readers/nanonis_sxm.py:22-39`
**Problem**: When the payload contains fewer planes than `DATA_INFO` advertises, the reader issues a `warnings.warn(UserWarning)` and returns the truncated list. `read_sxm` calls `sxm_plane_metadata(hdr, len(planes))`, so the resulting `Scan` reports `n_planes=truncated_count`. Nothing in `Scan.header` or `scan.experiment_metadata` records the truncation — provenance and exports look identical to a complete scan with fewer channels. The `UserWarning` may be filtered to silence in production logs.
**Why it matters**: A user processing a half-written SXM (incomplete acquisition, partial download) gets a `Scan` that looks correct but is silently missing a backward plane. Forward-backward blending produces nonsense.
**Suggested fix**: Attach a structured `truncation` record into `scan.header` (or `Scan.warnings`) so it survives into PNG/PDF/JSON exports and the GUI can display a banner. Keep the runtime warning too.

### 11. [S2] Createc VERT `_split_data_section` EOL detection only checks bytes immediately after DATA marker
**Location**: `probeflow/io/readers/createc_vert.py:282`
**Problem**: `eol = b"\r\n" if data_section[4:6] == b"\r\n" else b"\n"`. Only inspects two bytes immediately following the `DATA` marker to decide line ending. If a VERT file uses mixed EOL (e.g. legacy file ported between OSes), subsequent `find(eol)` either fails outright (raises `missing line ending after DATA marker`) or finds the wrong boundary and silently mis-parses the params line.
**Why it matters**: Cross-platform-transferred Createc VERT files can hit this; the failure mode is sometimes "wrong column header" rather than a clean error.
**Suggested fix**: Try `\r\n` first, fall back to `\n` based on which yields a parseable params line.

### 12. [S2] `_parse_string_data` (RHK SM4) silently truncates malformed string blocks
**Location**: `probeflow/io/readers/rhk_sm4.py:522-545`
**Problem**: When a string's declared length exceeds the remaining payload (line 540 `if pos + char_bytes > len(payload): break`), the loop breaks silently. The collected strings are positionally zipped with `_STRING_KEYS`, so a truncation loses *all* subsequent keys (e.g. `z_units`, `x_label`, `status_channel_text`). No warning is recorded.
**Why it matters**: A corrupt SM4 file with one bad string field silently drops `z_unit` for the affected page; `plane_units` then defaults to empty, and downstream unit conversions silently assume metres.
**Suggested fix**: When the loop breaks early, log a warning into the parser-notes list (already plumbed through). Optionally pad missing positions with empty strings so the positional mapping stays valid.

### 13. [S2] `_parse_pages` PAGE_INDEX_ARRAY stride assumes fixed per-page object record size
**Location**: `probeflow/io/readers/rhk_sm4.py:382` (`stride = PAGE_INDEX_ARRAY_SIZE + max(object_count, 0) * OBJECT_SIZE`)
**Problem**: The stride is hardcoded to `32 + object_count * 12`. The file-level `object_field_size` is checked at line 161-162, but there is no per-page check. SM4 files written by newer firmware with a different per-page object-record size would compute the wrong stride for every page beyond the first.
**Why it matters**: Newer RHK SM4 files may fail to load all pages; the loader could either silently skip pages or load garbage offsets.
**Suggested fix**: Cross-check stride against the page-index entry's own declared object-field size if present. Add an integration test against an SM4 file from a recent RHK firmware revision.

### 14. [S2] Nanonis SPM `np.loadtxt` chokes on comma-as-decimal-separator
**Location**: `probeflow/io/readers/nanonis_spec.py:114`, mirror case `probeflow/io/readers/createc_vert.py:333`
**Problem**: `np.loadtxt(StringIO(clean), dtype=np.float64, delimiter="\t")`. European-locale Nanonis software can save numeric values with comma decimal separator (`1,234e-9` instead of `1.234e-9`). `np.loadtxt` raises `ValueError` on the first such row, which the outer try/except re-wraps. The Createc DAT→SXM converter already proactively `.replace(",", ".")` for the rotation header (createc_dat_to_sxm.py:224) — the issue is known in the codebase but not consistently handled.
**Why it matters**: International users get hard failures on files saved on locale-specific machines with no hint that the cause is decimal separator.
**Suggested fix**: Detect comma-decimal in the first parsed row and re-parse with `.replace(",", ".")`, or preprocess unconditionally.

### 15. [S2] CSV writer header line is whitespace-separated and lacks structured channel-id / processing-state hash
**Location**: `probeflow/io/writers/csv.py:66-73`
**Problem**: Header is built as `f"plane={name} units={unit} Nx={Nx} Ny={Ny} width_m=... height_m=... source=..."`. With plane names that contain spaces (and Nanonis names sometimes do), a downstream parser cannot robustly split key=value pairs. No structured channel index or processing-state hash is written into the CSV itself; provenance lives only in the sidecar JSON.
**Why it matters**: CSV is the most-likely artefact a user mails to a collaborator. Loss of structured provenance on the CSV alone — once decoupled from its sidecar — is a recovery risk.
**Suggested fix**: Emit one `# key=value` comment per key, with quoted values for strings that may contain whitespace, and include `processing_state_hash`.

### 16. [S2] PDF writer y-axis tick labels inverted relative to image data
**Location**: `probeflow/io/writers/pdf.py:108-119`
**Problem**: `imshow(..., origin='upper', extent=(0, w_m*1e9, 0, h_m*1e9))`. With `origin='upper'` matplotlib draws array row 0 at the top of the canvas. The extent argument is `(left, right, bottom, top)`, so `bottom=0, top=h_nm` places the y-axis origin at the bottom of the canvas. Net effect: pixel at the top of the canvas (row 0, the first STM scan line acquired) sits at the labeled y-tick `h_nm`, but the user expects "row 0 = y=0" or at least the convention to be visible.
**Why it matters**: A reader of the PDF measuring distances from the y-tick labels does so against an axis whose direction is opposite to the data. Step-height callouts read directly off the PDF (without knowing the convention) are inverted.
**Suggested fix**: Use `extent=(0, w_nm, h_nm, 0)` to make the y-axis increase downward (matching `origin='upper'`), or switch to `origin='lower'` and pre-flip the array. Pick one convention and document it.

### 17. [S2] Createc VERT `_summarise_numeric_table` raises on any column-count mismatch
**Location**: `probeflow/io/readers/createc_vert.py:366-370`
**Problem**: The streaming summariser used by `read_createc_vert_report(include_arrays=False)` raises `ValueError` immediately when any row has a different column count than the first. Some Createc VERT variants pad early rows or omit trailing zeros. The full-parse path (`_parse_numeric_table` using `np.loadtxt`) is more permissive — so the metadata fast path crashes on files that the full read handles silently.
**Why it matters**: Folder indexing (`read_spec_metadata`) crashes on Createc VERT files with ragged columns, while opening the same file via `read_spec_file` succeeds. Inconsistent failure modes.
**Suggested fix**: Unify on the `np.loadtxt`-based path, or relax the check to ">= expected_columns" with a warning.

### 18. [S3] Createc DAT reader does not propagate decode `warnings` tuple to the resulting `Scan`
**Location**: `probeflow/io/readers/createc_scan.py:54-100` (does not consult `report.warnings` or `report.is_partial_scan`)
**Problem**: `read_createc_dat_report` populates a `warnings` tuple flagging trimmed rows, ignored trailing payload floats, partial scans, and channel-count fallback decisions. `read_dat` discards it — the `Scan` object has no field carrying these warnings. The `is_partial_scan` flag from the report is also dropped despite being explicitly documented in `docs/createc_dat_reader.md` as a value callers can use to distinguish completed from interrupted scans.
**Why it matters**: Partial scans (documented as the central reason for the `image_y_pos_max` machinery) should be visible in the GUI and in exports. Currently the partial-scan determination is made and then silently dropped.
**Suggested fix**: Add a `read_warnings` field to `Scan` (or store under `scan.header["_probeflow_decode_warnings"]`) and surface `is_partial_scan` similarly. Make `write_provenance_sidecars` carry it through.

### 19. [S3] RHK SM4 plane units never normalised — `pm`, `Å`, `nm` propagate raw
**Location**: `probeflow/io/readers/rhk_sm4.py:213, 477` (units returned verbatim from `strings.get("z_units")`); `_decode_image_payload:440-442` applies `z_scale` without unit conversion
**Problem**: RHK strings can be `m`, `pm`, `nm`, `Å`, `1` (dimensionless), or empty. The `physical_data = raw * z_scale + z_offset` line uses `z_scale` verbatim from the page header, and the Scan's `plane_units` is set to whatever string the file carried. If RHK firmware encoded `z_units='pm'` and the resulting `physical_data` is in *picometres*, downstream code that assumes Z is in metres (PDF scalebar, lattice extraction in `m`, `scan_range_m`-based pixel pitches) produces results off by 10^12. No unit normalisation step exists inside the reader.
**Why it matters**: Cross-vendor comparisons silently scale wrong. The severity depends on what real RHK exports populate the `z_units` string as — if always `m`, this is moot; if `pm` is common, it is S1.
**Suggested fix**: Map known unit strings (`pm`, `nm`, `Å`/`A`) to SI, multiply `z_scale` accordingly, and rewrite `plane_units` to canonical SI. Add a test fixture with a non-`m` z_unit.

### 20. [S3] `write_gwy` writes no provenance sidecar; provenance only embedded inside `.gwy`
**Location**: `probeflow/io/writers/gwy.py:75-152`
**Problem**: Unlike `write_sxm`, `write_pdf`, `write_png`, `write_csv` — all of which call `write_provenance_sidecars` — `write_gwy` only embeds provenance inside the `GwyContainer`. There is also no `check_provenance_sidecar_collisions` call. A Gwyddion round-trip (open + re-save) drops unknown metadata keys, silently losing provenance.
**Why it matters**: Provenance survives the immediate export but not a subsequent Gwyddion edit.
**Suggested fix**: Mirror the pattern from `write_pdf`/`write_csv`: write a `.gwy.probeflow.json` sidecar alongside.

### 21. [S3] `write_csv` uses `np.savetxt` with implicit text-mode newlines
**Location**: `probeflow/io/writers/csv.py:72-73`
**Problem**: `np.savetxt` opens the file in text mode on Windows, so universal-newline translation can produce mixed `\n` and `\r\n` separators depending on the writer's mode. The result on Windows is occasionally a CSV that some downstream parsers handle inconsistently.
**Why it matters**: Cross-platform polish. Not data-corrupting.
**Suggested fix**: Open the file explicitly in binary mode with `newline="\n"` and pass the file handle to `np.savetxt`.

### 22. [S3] `_data_offset_in_file` cushion cache is misnamed — only the lengths are cached
**Location**: `probeflow/io/sxm_io.py:36-57`
**Problem**: The module-level `_POST_END_BYTES_CACHE` and `_PRE_PAYLOAD_BYTES_CACHE` cache the bytes, but the public helper `_cushion_tail_lens` returns only the *lengths*. Other code paths (e.g. `write_sxm_with_planes` which calls `load_sxm_layout` indirectly via `_write_from_dat` -> `load_layout_and_format`) read the cushion files again from disk. The cache name suggests bytes are cached for reuse, but no read path reuses them.
**Why it matters**: Polish; no correctness impact.
**Suggested fix**: Either truly cache and return bytes, or rename to `_cushion_tail_lens_cached`.

## Cross-references
- Findings 7, 12, 13, 19 (RHK SM4 robustness) are the first IO-side coverage of the SM4 reader; prior reviews did not touch it.
- Finding 5 (DAT→SXM `SCAN_DIR='up'` mishandled on writeback) overlaps with image-processing review finding 4 (state pipeline does not update `scan_range_m` for shape-changing ops); both are silent orientation/metadata mismatches.
- Finding 10 (truncated SXM payload silently shortens plane list) parallels physics finding 3 / numerical finding 3 about data-driven defaults that hide problems from the user.
- Findings 15 (CSV provenance) and 20 (gwy provenance) form a pair: writers that leak structured context once decoupled from their JSON sidecars.
- Finding 2 (ROI sidecar `with_suffix`) is novel and high-impact for Createc users specifically — none of the prior reviews touched the sidecar path machinery.
- Finding 19 (RHK unit normalisation) is the most likely to silently introduce wrong-units bugs in real-world cross-vendor work; flagged S3 here only because the failure depends on a file whose `z_units` is not `m`. A real RHK fixture would let it be promoted to S1.

## Items consciously *not* flagged
- `_split_createc_dat_payload` correctly validates zlib decompression at each candidate `DATA` byte — robust against header-text false positives.
- `parse_sxm_header` byte-mode reading is correct for latin-1 headers (errors='replace' keeps progress).
- `_decode_image_payload` defensively tries zlib decompression when raw size doesn't match — robust against compressed payloads.
- `roi_sidecar_candidates` list of fallbacks is broad enough that an existing sidecar will usually be found despite finding 2.
- `write_json` uses temp-file-then-rename atomic writes — good.
- `save_roi_set_sidecar` uses temp-file-then-rename — good.
- `_first_column_diagnostics` correctly handles empty `finite_first` arrays — no NaN traps.
- `scan_range_m_from_header` units (Å → m) are explicit and consistent with `construct_hdr`.
- Createc DAT reader correctly uses `<f4` (little-endian float32) explicit dtype — robust to host endianness.
- Nanonis SXM reader correctly uses `>f4` (big-endian float32) explicit dtype — robust to host endianness.
- RHK SM4 reader uses explicit little-endian struct codes (`<`) throughout — robust to host endianness.
