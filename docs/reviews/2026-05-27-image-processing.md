## Summary
- 16 findings: S0=0, S1=6, S2=8, S3=2
- Top 3 concerns:
  1. Pipeline ordering hard-codes `align_rows → plane_bg` (and forwards none of the calibration metadata), so the polynomial fit is performed against rows whose median has already been subtracted and the surface model is effectively rebuilt against zero-mean rows — silently weakening multi-step background correction (S1).
  2. `apply_processing_state` never forwards `pixel_size_x_m / pixel_size_y_m` to `subtract_background(step_tolerance=True)` or to `facet_level`, so the user-facing slope threshold (`step_threshold_deg=3°`) is interpreted in *data-units-per-pixel* instead of physical rise/run on every scan. The defaults were carefully written to make the angle physically meaningful — that calibration is dropped on the GUI path (S1).
  3. `quantize_bit_depth` runs *before* `arithmetic_ops` in the GUI adapter, so any constant add/subtract after Image > Type > 8-bit produces values that are no longer at integer-multiple level boundaries — the "256 distinct values" promise of 8-bit is broken whenever a subsequent arithmetic step is appended (S1).

## Findings

### 1. [S1] Calibration metadata silently dropped from `subtract_background(step_tolerance=True)` and `facet_level` on the GUI path
**Location**: `probeflow/processing/state.py:530-538` (subtract_background) and `559-563` (facet_level); see `probeflow/processing/background.py:497-503, 591-599` and `probeflow/processing/alignment.py:64-116` for the receivers
**Problem**: `subtract_background` and `facet_level` both accept `pixel_size_x_m`/`pixel_size_y_m` so the `step_threshold_deg` comparison is a physical rise/run ratio. `apply_processing_state` calls both without passing those arguments, so the defaults (1.0 m/px) take effect every time. With typical 50 pm/px scans, the gradient threshold `tan(3°)≈0.052` corresponds to **about 0.5 nm/pixel** instead of the documented "3° slope on the surface": the step-tolerant fit treats nearly every real step edge as flat. The GUI/CLI silently degrade to a non-step-tolerant fit even when the user explicitly enabled it.
**Why it matters**: Stepped surfaces (Au(111), Si(111)) — the canonical use case for step-tolerant background — get a tilted plane biased by the step edges. The user enabled the option and saw no warning; the image is wrong but plausible. This affects every "facet_level" and `plane_bg + step_tolerance` invocation through the GUI/state.
**Suggested fix**: Pipe pixel sizes through `apply_processing_state` (the function already has the array shape; the scan's `scan_range_m` must be passed in as a new optional arg). Pass them to both `subtract_background(..., pixel_size_x_m=..., pixel_size_y_m=...)` and `facet_level(..., pixel_size_x_m=..., pixel_size_y_m=...)`.

### 2. [S1] `quantize_bit_depth` is appended before `arithmetic_ops`, breaking the "discrete level" guarantee
**Location**: `probeflow/processing/gui_adapter.py:225` (geometric_ops loop, includes quantize) → `:326` (arithmetic_ops loop). The supported quantize call appears at `gui_adapter.py:321-324`.
**Problem**: Pipeline order is `... → geometric_ops → arithmetic_ops`. If a user runs Image > Type > 8-bit then later subtracts a constant (or any image-arithmetic operand), the constant offset adds a non-integer-multiple shift to every pixel. The result still has 256 distinct values, but they are no longer evenly spaced at the quantum used by the encoding step, so any downstream tool that assumes "integer level encoding" (e.g. histogram analysis, PNG/TIFF export at 8-bit, comparison with a second quantized image) breaks. Cross-ref: numerical agent finding 3 (data-dependent vmin/vmax) is about reproducibility; this is an orthogonal issue — *order* of operations.
**Why it matters**: 8-bit conversion is typically the *last* step before save; placing it inside the geometric block (which is followed by arithmetic) hides this. There is also no GUI control to constrain step ordering — the user has no way to say "this comes last".
**Suggested fix**: Either (a) emit `quantize_bit_depth` as its own block executed last in the gui_adapter, after `arithmetic_ops`; or (b) reject arithmetic steps appended after a quantize step in the same processing state with a clear warning ("Arithmetic after 8-bit quantization will produce non-discrete levels — remove the quantize step or the new arithmetic step").

### 3. [S1] `align_rows` mutates the rows that `subtract_background` then fits — feature-amplitude bias
**Location**: `probeflow/processing/state.py:523-538` (sequence); `probeflow/processing/alignment.py:34-46` (median/mean subtraction); `probeflow/processing/background.py:520-616` (polynomial fit)
**Problem**: The canonical GUI order is `align_rows(median)` → `plane_bg(order=1)`. After median row subtraction, every row has median = 0. A subsequent global plane fit (order 1) on a heavily row-aligned image is dominated by the global column trend — the row-direction tilt is no longer present, and the plane intercept is pulled toward zero. Sequenced the opposite way (plane first, then per-row median), the row alignment correctly removes the residual offsets the plane could not capture. The current order silently approximates "tilt correction" with "column-direction tilt only" because row tilt has been zeroed out.
**Why it matters**: For STM scans with slow-axis thermal drift (very common), the user expects align_rows to handle row-to-row offsets and plane_bg to handle global tilt. The current implementation effectively gives them only a column-direction tilt correction. Step-height metrology on stepped surfaces is biased — the global plane fit can no longer compensate for the slow-axis offset trend.
**Suggested fix**: Swap the canonical order to `plane_bg` → `align_rows`, or document that `align_rows` should follow `plane_bg`. Cross-ref: numerical finding 7 (NaN restoration in align_rows) hits the same code path.

### 4. [S1] `apply_processing_state` does not update `scan_range_m` when `scale_image`/`rotate_arbitrary`/`shear`/`affine_lattice_correction` change the array shape
**Location**: `probeflow/processing/state.py:702-730` (rotate/shear/scale_image execution) vs. `:816-821` (only the geometric helper updates `scan_range_m`, and only for rot90/rot270)
**Problem**: `apply_geometric_op_to_scan` updates `scan.scan_range_m` for rot90/rot270, but `apply_processing_state` never gets a scan handle — it only mutates the 2-D array. When the processing pipeline runs `scale_image(new_height=512, new_width=512)` on a 256×256 plane, the output array is 512×512 but the GUI's stored `scan_range_m` is still the original 256×256 worth of physical extent. Display scale bars, FFT k-axes (`pixel_size = scan_range_m / Nx`), feature pixel-to-nm conversions, and saved provenance all use a now-wrong pixel size. The same hazard applies to `rotate_arbitrary` (`reshape=True` enlarges canvas), `affine_lattice_correction(expand_canvas=True)`, and `shear(expand_canvas=True)`.
**Why it matters**: Every downstream measurement after a non-90°-rotation pipeline step is in the wrong physical units. The scale bar on the exported PNG is wrong by the same factor. This is silent — no exception, no warning.
**Suggested fix**: Two options. (a) Have `apply_processing_state` return the new physical pixel size alongside the array, and update `scan.scan_range_m` whenever the array shape changes. (b) Force the user-facing transforms that change shape (`scale_image`, `rotate_arbitrary` with `reshape=True`, `affine_lattice_correction` with `expand_canvas=True`, `shear`) to commit their changes through `apply_geometric_op_to_scan` instead of through the state pipeline. Cross-ref: physics finding 2 (rotate NaN halo) — fixing this would naturally hold scan_range_m correct as well.

### 5. [S1] `fft_magnitude` does not restore NaN mask; non-rectangular ROI FFT cannot be told apart from ROI=full-image visually
**Location**: `probeflow/processing/filters.py:426-470`
**Problem**: NaN pixels in the input are mean-filled in step 2 (`crop = np.where(nan_mask, mean_val, crop)`), then never restored in the output magnitude. The magnitude is real-positive everywhere; pixels that *should* be flagged as "no valid data after ROI mask + NaN substitution" are silently replaced with a finite spectrum. Because the GUI displays this as `log1p(|F|)`, the user has no signal that the magnitude on the boundary or outside the ROI is partly synthetic. Cross-ref: physics finding 1 (biased DC subtraction on masked ROI) shares the same code path.
**Why it matters**: Quantitative work (peak picking, Bragg-radius extraction, q-space ellipse fit) on the masked-ROI FFT silently consumes the synthesized signal at the boundary. Edge artifacts from the mask discontinuity are real spectral leakage but the user reads the resulting peaks as physical.
**Suggested fix**: Build a "valid pixel" mask propagated alongside the FFT. The output magnitude has no NaN convention (it's a spectrum, not an image of the same domain), but at minimum mark the boundary of the ROI's "valid" extent in the returned axes metadata — or always run the magnitude with explicit window correction (see physics finding 5) so the spurious spectrum is at the proper coherent-gain level.

### 6. [S1] `set_zero_plane` fits the plane in raw pixel-index coordinates → ill-conditioned for large images
**Location**: `probeflow/processing/geometry.py:224-233`
**Problem**: The fit builds the 3×3 design matrix `A = [[x, y, 1.0], ...]` where x, y are **raw pixel indices**, not normalised. For a 1024×1024 scan the matrix has condition number ~10⁶, vs. ~1 if normalised to [-1, 1]. The 3-point linear solve usually stays well within float64, so this isn't catastrophic in 64-bit. **However:** the design matrix is also rank-checked with `np.linalg.matrix_rank(A) < 3` at integer-pixel resolution, so two click points that round to the same pixel slip through (matrix_rank uses a default tolerance proportional to the largest singular value, which is dominated by Nx). When users click sub-pixel-close points, the resulting plane is over-conditioned but accepted, producing a tilted "zero plane" whose slope is dominated by the small noise difference between two near-coincident patches.
**Why it matters**: Manual "Z=0 by 3 clicks" workflow on a small atomic image (e.g. 64×64) is a key STM operation. Two clicks on adjacent atoms produce a very large slope and an unphysical tilt that the user must redo. The rank-check could be tightened with a physical "3-click triangle must enclose ≥ N pixels" guard.
**Suggested fix**: Normalise coordinates to [-1, 1] (consistent with `subtract_background`), then either re-scale coefficients back or evaluate the plane in normalised space. Add a separate "triangle area >= 4 pixels" guard to catch sub-pixel-close click pairs.

### 7. [S2] `quantize_bit_depth` operates on data after geometric ops, so its level granularity becomes shape-dependent
**Location**: `probeflow/processing/state.py:740-741`; cross-ref `probeflow/processing/geometry.py:641-678`; ordering at `probeflow/processing/gui_adapter.py:225-324`
**Problem**: Because quantize runs late in the pipeline, the data extent it sees has already been changed by `align_rows`, `plane_bg`, geometric ops (rotation/shear NaN halo), etc. After `rotate_arbitrary` and `affine_lattice_correction`, the array contains NaN bands at the canvas extremes (cross-ref physics finding 2). The quantization uses `arr[finite].min/max`, so any tip-crash row that happened to survive the prior steps still defines vmin/vmax (cross-ref physics finding 3 and numerical finding 3 for the orthogonal reproducibility angle). On top of that, the new vmin/vmax now reflect post-rotation interpolated values, not the original calibration.
**Why it matters**: Two identical scans with slightly different cropping or rotation angles get different bit-depth-quantized values. The quantized image inherited "step quanta" of, e.g., 4 pm in one and 4.3 pm in the other; comparing them is meaningless without explicit per-image quantum metadata, and the quantization step does not record `vmin`/`vmax`/`quantum` in `step.params`.
**Suggested fix**: Record the realised `vmin`, `vmax`, and `quantum` (= (vmax-vmin)/(2**bits-1)) in the step's params dictionary at the moment quantize is applied, so future re-runs are reproducible.

### 8. [S2] `affine_lattice_correction` interpolation contaminates valid pixels via NaN fill, then OOB-mask drops them differently from rotate_arbitrary
**Location**: `probeflow/processing/geometry.py:286-408`, especially `:352-356` (mean fill before interpolation) and `:404-406` (OOB mask)
**Problem**: The function mean-fills NaN pixels for `map_coordinates`, then masks "out-of-bounds" output pixels back to NaN only where the *inverse map* falls outside the input bounds. This correctly handles canvas expansion, but does **not** restore the original NaN pixels in the *interior* of the image: an isolated NaN island in the source becomes a small disk of interpolated synthetic data after rotation. Compared with `rotate_arbitrary` (physics finding 2, uses `cval=NaN` so a halo of NaN expands), the two transforms produce visibly different NaN topologies on identical input. The user cannot predict which fill policy will apply.
**Why it matters**: A scan with a streak of tip-crash NaN gets a clean rotation through `affine_lattice_correction` (synthetic data fills the streak) but a wide NaN halo through `rotate_arbitrary`. Lattice correction provenance therefore depends on *which path* the user clicked. Downstream analysis using `np.isfinite` masks (e.g. roi_statistics) sees different valid-pixel counts.
**Suggested fix**: Standardise: build a NaN-mask through the same coordinate transform with `order=0`, then `out[transformed_nan_mask] = np.nan` in both functions.

### 9. [S2] `fft_magnitude` window choice and `fourier_filter` window choice diverge silently
**Location**: `probeflow/processing/filters.py:73-82` (fourier_filter) vs. `:438-456` (fft_magnitude)
**Problem**: `fourier_filter` accepts `window ∈ {hanning, hamming, none}`; `fft_magnitude` accepts `window ∈ {hann, tukey, none}` — note "hanning" vs "hann" (treated as different strings), and `fourier_filter` has no Tukey while `fft_magnitude` has no hamming. The same user-facing label drives different code paths. A workflow that does "FFT viewer with Tukey, then apply FFT filter in pipeline" cannot use a consistent window choice. The 2-D Tukey assembly in `fft_magnitude` (lines 444-454) is also asymmetric for `n < 2 * edge`: when N is small (e.g. 32-pixel ROI) and `alpha=0.25` the ramp `edge = round(alpha/2 * N) = 4`, so the first/last 4 pixels are tapered but the rest is flat — fine, but the docstring suggests `alpha=0.25` means "25 % plateau", actually it means "12.5 % ramp on each side". Cross-ref physics finding 6 (np.hanning zero-endpoint) compounds this.
**Why it matters**: Visual FFT in viewer and applied filter use mismatched window normalisation, so a user "tuning by eye" sees a different spectrum from the one actually being filtered. Tukey alpha semantics in the docstring don't match scipy's definition.
**Suggested fix**: Centralise on `scipy.signal.windows.{hann, hamming, tukey}` in both functions, use the standard alpha semantics (`alpha` = fraction of cosine taper, half on each side), and accept the same string set in both.

### 10. [S2] `_image_utils._nan_normalized_gaussian` uses `mode='reflect'` — boundary smoothing leaks across the image
**Location**: `probeflow/processing/_image_utils.py:37-55`
**Problem**: Both the values and the weights are reflected at the boundary in the NaN-normalised gaussian. For an image with a large NaN strip (e.g. cropped scan-line, tip-retracted top row), the reflected weights make boundary pixels appear "well sampled" even though the only finite data is on one side. The blurred result close to the boundary is biased toward the interior reflection. This is the kernel that `gaussian_smooth` and `gaussian_high_pass` both build on (lines 174-190, 106-121 in `filters.py`).
**Why it matters**: STM scans often have systematic boundary anomalies (first/last few rows during tip approach). A `gaussian_high_pass` over those rows reflects interior signal back and overstates the high-pass response there. The user perceives this as "the high-pass filter found something at the edges" — actually an artefact.
**Suggested fix**: Use `mode='nearest'` (or `mode='constant', cval=0` and rely on the weight normalisation) so the weight at the boundary correctly reflects "no data outside".

### 11. [S2] Geometric flips/rotations do not transform the auxiliary scan metadata (forward/backward channel naming) — only the array and ROI
**Location**: `probeflow/processing/state.py:796-841` (apply_geometric_op_to_scan)
**Problem**: For multi-channel scans (forward + backward) the convention is that the backward channel is left-right mirrored relative to the forward channel (cf. `blend_forward_backward` at `geometry.py:415-441`). After `flip_horizontal` on the *whole scan*, the convention silently inverts: what was "forward" is now mirrored, but the channel name is unchanged. There is no metadata flip mirroring the array flip. The user later runs `blend_forward_backward(fwd, bwd)`, which still flips `bwd` left-right — now incorrectly because `fwd` was already mirrored by the earlier `flip_horizontal`.
**Why it matters**: Multi-channel STM workflows that include a global flip get a doubly-flipped backward channel without warning. Forward-backward blending produces obvious ghosting that the user will (correctly) attribute to mis-alignment, but the cause is silent metadata mismatch, not detector physics.
**Suggested fix**: Either (a) explicitly invert the "forward" semantic in `scan.plane_names` / `scan.plane_directions` whenever `flip_horizontal` is applied, or (b) refuse `flip_horizontal` on multi-channel scans without an explicit confirmation.

### 12. [S2] `line_profile` `width_px=1.0` cannot be replicated reliably across calls — boundary rounding makes the perpendicular swath asymmetric
**Location**: `probeflow/processing/geometry.py:107-132`
**Problem**: The width=1 branch (line 107-109) uses `map_coordinates` directly, no swath. The width>1 branch uses `n_perp = int(round(width_px))` perpendicular samples. So `width_px=1.0` → n_perp=1 (no swath, even though width>1 branch isn't taken). `width_px=1.5` → falls into width>1 branch, `n_perp = int(round(1.5)) = 2`, two perpendicular samples 0.25 pixel either side of the centre. `width_px=2.5` → `n_perp=3` (rounding); two samples symmetric at ±0.75, plus the centre? No — three samples at -0.75, 0, +0.75. Compare `width_px=2.4`: `n_perp=2` (rounds down), so two samples at ±0.35. The swath density jumps abruptly between 2.4 and 2.5 — twice the sample density per unit width at 2.5. Cross-ref numerical finding 11 (reflect mode bias at borders).
**Why it matters**: The line-profile widget in the GUI lets the user set `width_px=2.5` via a slider with 0.1 px steps. The user sees a sudden quality change at 2.5/3.5/... boundaries that should not exist.
**Suggested fix**: Use a fixed sample density (e.g. 4 perpendicular samples per pixel of width), independent of `n_perp = round(width_px)`. Document the minimum-step that produces a visible change.

### 13. [S2] Threshold dialog highlight pixmap uses RAW vmin/vmax (data extremes) for grey, not the display percentiles
**Location**: `probeflow/gui/dialogs/threshold_dialog.py:300-336`, especially `:310-313`
**Problem**: The "highlight pixmap" preview path renders the image in greyscale by normalising to `[self._vmin, self._vmax]`, which are the **data extremes** (`finite.min()` / `finite.max()` on lines 147-148). The main image viewer normalises to the 1–99 percentile clip. So the user-facing histogram in the dialog (which itself uses 0.1–99.9 percentiles) and the highlight pixmap show **different greys**: a single outlier pixel can wash the highlight pixmap out to near-uniform black/white, while the rest of the dialog shows a properly-clipped histogram. Then the user picks a "lo/hi" by visual inspection and the highlights they see are right-but-the-greyscale-context is wrong.
**Why it matters**: The dialog's whole UX is "visually pick the threshold by looking at outside-the-range pixels highlighted in red". If a single bright outlier dominates the greyscale, the user cannot see the surface, and the highlight has no context. This is exactly the same root issue that physics finding 3 and numerical finding 3 flag for quantize_bit_depth — data extremes vs. robust clip range.
**Suggested fix**: Use the same `clip_range_from_array(arr, 1.0, 99.0)` as the rest of the viewer to set the greyscale range. Make the highlight overlay use the user-chosen `[lo, hi]` only for the *coloured* overlay, never for the greyscale background.

### 14. [S2] `_modal_shift` falls back to mid-bin when no pixel "exactly" lies in the modal bin, but doesn't propagate the histogram's discreteness assumption
**Location**: `probeflow/processing/background.py:623-648`; consumed by `stm_line_background` at `:651-707`
**Problem**: After `np.histogram(values, bins=...)`, the function looks for `(values >= edges[peak]) & (values <= edges[peak + 1])` (line 645). `np.histogram` puts values in bins with right-open convention (`[lo, hi)`) except for the last bin which is closed (`[lo, hi]`). The `&` predicate uses `<=` on both sides, so it can include pixels that `np.histogram` placed in the *next* bin. Worst case: if there are only two distinct values and the histogram has many bins, the "modal bin" picked by `argmax` can be empty under the predicate (because of float jitter at the edge), and the function returns the mid-bin centre — which is a value that doesn't exist in the data. Cross-ref numerical finding 6 (same observation from a different angle).
**Why it matters**: For STM-style scan-line background with `stm_line_background`, the modal shift is propagated row-to-row. A single bad row that pushes prev_shift to a synthetic mid-bin value drifts the entire downstream scan height baseline.
**Suggested fix**: Switch to `np.searchsorted(edges, values, side='right') == peak + 1` (as numerical finding 6 also suggests) to match np.histogram's binning convention exactly.

### 15. [S3] `threshold_image(mode='binarize')` returns float64 1.0/0.0 instead of bool/uint8
**Location**: `probeflow/processing/geometry.py:621-638`
**Problem**: When binarizing the image, the function returns `mask.astype(np.float64)` (line 637). This is wasteful (8× the memory of a bool, 8× the memory of uint8) and clobbers the original NaN mask: pixels that were NaN become 0.0 (since `lower is not None and a >= lower` is False for NaN). The clip mode does preserve NaN explicitly (lines 626-630), so the two modes have inconsistent NaN semantics — binarize silently turns NaN into "below threshold". For downstream display, the binarize result then re-enters the histogram pipeline where 0.0 looks like a real height value.
**Why it matters**: A binarized mask of a scan with tip-retracted NaN strips paints those strips black (0). The user may want them to remain NaN (i.e. distinct from "below threshold"). The result is also 8× larger than necessary.
**Suggested fix**: Return `mask.astype(np.uint8)` (or document the float convention) and explicitly write NaN back into the binarize output where the input was non-finite.

### 16. [S3] `gui_adapter.processing_state_from_gui` silently squashes failed coercions (no warning to the user)
**Location**: `probeflow/processing/gui_adapter.py:178-184, 213-219, 226-244, 326-339`
**Problem**: Every `try/except` for malformed parameters (`peaks`, `set_zero_plane_points`, `geometric_ops[i].matrix`, `arithmetic_ops`) silently continues on failure. A typo in a saved processing-state JSON drops the step without telling the user. The same processing applied via CLI on a different platform with different JSON parsing might lose different steps. Cross-ref: numerical finding 14 (`apply_outlier_mask` similarly silent on bad input).
**Why it matters**: A user shares a `.processing.json` sidecar; the receiver loads it and gets a different image because two steps were silently dropped. There is no audit trail.
**Suggested fix**: Replace `continue` with a `warnings.warn(...)` (UserWarning) so the CLI and GUI see a notification. Even better: collect the errors and surface them to the export-provenance layer.
